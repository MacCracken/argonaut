## [MEDIUM-1] `fork_exec_service` child inherits PID-1 signal mask

- **File:** `src/process_mgmt.cyr` `fork_exec_service` child
  path (post-fork, pre-execve).
- **Behaviour:** the PID-1 supervisor loop in `src/main.cyr`
  blocks SIGTERM / SIGINT / SIGCHLD via
  `sys_sigprocmask(SIG_BLOCK, ...)` so the signalfd path can
  drain them. The block is per-process; a fork inherits the
  signal mask, and `execve` preserves it (POSIX). Services
  spawned by `fork_exec_service` therefore start with all three
  signals **blocked**. A daemon that registers a SIGTERM
  handler for graceful shutdown would see no SIGTERM delivered
  even when argonaut forwards one — the kernel queues it
  against the masked process indefinitely.
- **Impact:** silently breaks shutdown for any service that
  relies on default signal disposition. Pre-1.6.2 argonaut
  didn't have a supervisor loop, so the issue didn't exist.
  Lit up the moment 1.6.2 added the sigprocmask + signalfd
  block. **MEDIUM** — services in the default registry happen
  not to rely on SIGTERM yet (they're all `STATE_STOPPED` until
  an explicit start), but the latent breakage shows up the
  first time a real service is wired.
- **Fix (shipped 1.6.3):** `fork_exec_service` child path
  unblocks the full signal set before `sys_execve` via
  `sys_sigprocmask(SIG_SETMASK, &empty_set, 0)`. The child
  then has the default disposition, free to install its own
  handlers.
- **Regression coverage:** `tests/tcyr/audit_findings.tcyr`
  `closeout-medium1-fork-exec-sigmask-reset` group asserts
  the new `reset_child_signal_mask()` helper (extracted from
  the inlined fix) issues `SYS_RT_SIGPROCMASK` with
  `SIG_SETMASK = 2` and a zeroed mask.

## [MEDIUM-2] `fork_exec_service` empty envp drops PATH

- **File:** `src/process_mgmt.cyr` `fork_exec_service` child
  path; the 1.6.2 rewrite hardcoded an empty envp.
- **Behaviour:** services spawned by `fork_exec_service`
  receive an empty environment. Most service binaries probe
  `PATH` to locate dependencies (e.g. `/usr/bin/python3`
  needs `PATH` to find `gcc` for native extensions; daemons
  shelling out for setup scripts need `PATH`). An empty
  envp means execve succeeds (binary loaded) but downstream
  PATH-based lookups fail.
- **Why this happened:** the 1.6.2 fix for the
  nested-fork-via-`exec_env_str` bug noted that
  `svc_def_env` returns a map (not a vec) and the pre-1.6.2
  `exec_env_str(argv, env)` read garbage when calling
  `vec_len(map)`. The fix punted on the map→flat-cstrs
  conversion and passed an empty envp.
- **Impact:** services lose PATH (and any other env). For
  argonaut's current default-service set this is latent (no
  service is auto-started yet), but the first real service
  wiring would hit it. **MEDIUM** — silent functional
  degradation.
- **Fix (shipped 1.6.3):** pass a minimal envp containing
  the standard PATH used by service init systems:
  `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`.
  The full `svc_def_env` map → flat-cstrs conversion (so
  services can extend the env) tracks as a separate roadmap
  item; this fix gives every service a usable PATH without
  needing config.
- **Regression coverage:** `closeout-medium2-fork-exec-envp-path`
  asserts the new `build_default_envp()` helper produces a
  vec whose first entry starts with `"PATH="`.

## [LOW-1] `pid1_harness_requested` substring match boundary

- **File:** `src/pid1_harness.cyr` `pid1_harness_requested`.
- **Behaviour:** scans `/proc/cmdline` for the literal
  substring `argonaut.harness=1`. A cmdline containing
  `argonaut.harness=11` matches (the prefix is identical),
  and `noargonaut.harness=1` also matches. Neither shape
  ever appears in production cmdline strings — the kernel
  parses space-separated tokens — but the substring search
  is over the raw byte buffer.
- **Impact:** none in practice. The harness flag is
  developer-controlled; nobody passes
  `noargonaut.harness=1`. **LOW** — defense-in-depth.
- **Fix (shipped 1.6.3):** require a boundary before the
  match (start-of-buffer, space, or tab) AND a boundary
  after (`=1` followed by end-of-buffer / space / tab /
  newline). Closes both edge cases.

## [LOW-2] /proc/cmdline read buffer capped at 1024 bytes

- **File:** `src/pid1_harness.cyr` `pid1_harness_requested`.
- **Behaviour:** reads up to 1024 bytes from /proc/cmdline.
  If a fleet-management system sets a kernel cmdline longer
  than 1024 bytes and places `argonaut.harness=1` past that
  boundary, the flag isn't seen.
- **Impact:** none today — Linux `CONFIG_CMDLINE_SIZE` is
  typically 2048 or 4096 bytes max, and argonaut deployments
  don't have very long cmdlines. **LOW** — documented;
  bump to 4096 when a consumer surfaces a long cmdline.
- **Disposition:** documented in this audit; no fix shipped.

## [LOW-3] dyn-loader bundling `ldd` parse robustness

- **File:** `qemu/build-initramfs.sh`.
- **Behaviour:** the 1.6.2 dyn-loader bundling step parses
  `ldd` output via `awk '/=>/ {print $3} /^\s*\//{print $1}'`.
  The pattern handles the two common `ldd` line shapes:
  `libfoo.so => /path/to/libfoo.so` and bare absolute paths
  (the dynamic linker itself). Lines for the kernel-provided
  vDSO (`linux-vdso.so.1`) and statically-resolved entries
  produce no third field and are silently skipped — correct
  behaviour. Lines containing spaces in paths (uncommon)
  would mis-parse.
- **Impact:** none on stock Arch / Debian / Ubuntu /
  Fedora. Theoretical edge case. **LOW** — documented.
- **Disposition:** documented; no fix shipped.
