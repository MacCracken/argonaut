# Argonaut Security Audit — 2026-04-26

P(-1) hardening pass before the **1.4.0** minor bump. Toolchain is
cyrius 5.7.5 / cc5 5.7.6, libro 2.0.5, patra 1.1.1. Baseline numbers
are `bench-history.csv` label `v1.4.0-pre-audit`. 26 test suites /
607 assertions / 0 failures going in.

This audit combines: external CVE / 0day research against the init
+ service-manager surface (systemd, OpenRC, s6, runit), 2025–2026
disclosures; argonaut-specific code review of process management,
health checks, sd_notify protocol handling, edge boot (dm-verity /
LUKS), and tmpfiles execution paths; and a defense-in-depth sweep
of the buffer / syscall / pointer surface against the cyrius 5.x
language gotchas (per `vidya/content/cyrius/field_notes/language.cyml`).

## External Threat Landscape

The init / service-manager class has had four directly relevant
disclosures in the 2025–2026 window. Each is mapped to argonaut's
exposure below.

### CVE-2025-4598 — `systemd-coredump` PID-recycling SUID coredump leak (HIGH)

- **Class:** TOCTOU race between kernel coredump invocation and userspace
  metadata read. Attacker forces a SUID process to crash and replaces
  it with a non-SUID binary at the same PID before `systemd-coredump`
  reads `/proc/$PID/auxv`. Result: privileged coredump (e.g.
  `/etc/shadow`-loaded process memory) becomes readable.
- **Argonaut exposure:** argonaut does **not** consume coredumps. It
  reads PID files (`read_pid_file` in `process_mgmt.cyr`), which is the
  same general pattern (PID-from-untrusted-source → operate on PID).
  See finding **M4** below.
- **Reference:** [Oracle Linux blog — Analysis of CVE-2025-4598](https://blogs.oracle.com/linux/analysis-of-cve-2025-4598), [Red Hat advisory](https://access.redhat.com/security/cve/cve-2025-4598).

### CVE-2026-3888 — `snap-confine` × `systemd-tmpfiles` LPE (HIGH)

- **Class:** Long-window TOCTOU between `systemd-tmpfiles` cleanup
  (10–30 day timer) deleting `/tmp/.snap`, and `snap-confine`
  bind-mounting that path on next sandbox init. Attacker recreates
  the directory with malicious payloads in the gap.
- **Argonaut exposure:** argonaut's `tmpfiles.cyr` generates
  tmpfiles.d-style commands but does **not** consume them. Real
  exposure is in `notify.cyr`'s `sys_unlink(socket_path)` +
  `sys_mkdir("/run/argonaut", 0o755)` sequence. See finding **H2**.
- **Reference:** [Qualys advisory](https://blog.qualys.com/vulnerabilities-threat-research/2026/03/17/cve-2026-3888-important-snap-flaw-enables-local-privilege-escalation-to-root), [The Hacker News writeup](https://thehackernews.com/2026/03/ubuntu-cve-2026-3888-bug-lets-attackers.html).

### CVE-2026-4105 — `systemd-machined` `RegisterMachine` LPE (MEDIUM)

- **Class:** improper access control on a privileged D-Bus method's
  `class` parameter, leaving an attacker-controlled machine object
  reachable.
- **Argonaut exposure:** argonaut has no D-Bus surface today. N/A.
- **Reference:** [GitHub Security Advisory](https://github.com/systemd/systemd/security/advisories/GHSA-4h6x-r8vx-3862), [Red Hat Bugzilla](https://bugzilla.redhat.com/show_bug.cgi?id=2447262).

### Nimbuspwn (CVE-2022-29799 / CVE-2022-29800) — historical class reference

- **Class:** directory traversal + symlink TOCTOU in
  `networkd-dispatcher`. Attacker replaces scripts argonaut-class
  daemons believe to be root-owned.
- **Argonaut exposure:** Argonaut runs `safe_cmd_*` with explicit
  argv (no shell), and `validate_device_path` rejects `..` in
  edge boot device args. Direct exposure low. The `notify_bind`
  unlink-then-mkdir sequence is the closest pattern — see **H2**.
- **Reference:** [Microsoft 365 Defender writeup via FourCore](https://fourcore.io/blogs/nimbuspwn-linux-vulnerabilities-allows-root-access-cve-2022-29799-and-cve-2022-29800).

### sd_notify spoofing (no specific CVE — protocol-level)

- **Class:** any process that can reach a service-manager's notify
  socket and is not credential-gated can fake `READY=1`,
  `MAINPID=<pid>`, `STATUS=...` notifications. systemd defends with
  the `NotifyAccess=` model (`SO_PEERCRED` check). PID 1 also sends
  VSOCK packets from a privileged port to address the converse
  attack of unprivileged guests sending malicious notifications to
  the host.
- **Argonaut exposure:** **direct hit**. See findings **H2** and
  **M1**.
- **Reference:** [`sd_notify(3)` — `NotifyAccess=` semantics](https://www.freedesktop.org/software/systemd/man/latest/sd_notify.html).

## Findings — Argonaut Source

Severity tags use the project convention: **CRITICAL** (remote /
priv-esc), **HIGH** (moderate effort), **MEDIUM** (specific
conditions), **LOW** (defense-in-depth). Each finding lands a
regression test in `tests/tcyr/` before the fix, per the
project's regression-tests-over-docs rule.

### [HIGH-1] `check_tcp_connect` and `HTTP_GET` ignore host argument

- **File:** `src/health.cyr` lines ~46–53 (TCP), ~190–197 (HTTP).
- **Behaviour:** the function takes a `host` / `target` URL but
  hardcodes the destination address to `127.0.0.1`:
  ```
  store8(&addr + 4, 127);
  store8(&addr + 5, 0);
  store8(&addr + 6, 0);
  store8(&addr + 7, 1);
  ```
  Any TCP/HTTP health check against a non-local host silently
  probes localhost instead. Worse, HTTP_GET also sends
  `Host: 127.0.0.1` in the request line.
- **Why this is security-relevant:** A health check that always
  reports "healthy" because some unrelated localhost service
  responds is a **trust anchor lying to the supervisor**. A failed
  external dependency wearing a green badge is exactly the
  pre-condition for cascading-failure scenarios (the supervisor
  refuses to restart, refuses to depend-on-ready a downstream).
  Severity: **HIGH** — not because of an attacker primitive, but
  because the safety property the function claims (this service
  responds at the configured endpoint) is unconditionally broken.
- **Regression:** `tests/tcyr/health_host_routing.tcyr` —
  `execute_health_check` against a non-localhost target with a
  closed port at 127.0.0.1 must NOT report passed. (For the test
  we use a target string that resolves to localhost only if the
  host arg is consulted, and assert the failure path when the host
  arg differs.)
- **Fix:** parse host out of `target` (TCP) and the URL (HTTP);
  resolve via `gethostbyname` or simple IPv4 dotted-quad parse;
  populate the `sin_addr` accordingly. Out of scope for the 1.4.0
  pass to add a full resolver — instead, accept `127.0.0.1` /
  `localhost` / `[::1]` literals, and **reject anything else with
  a clear error message** rather than silently probing localhost.
  This preserves the safety property at the cost of feature
  surface; full resolver lands as roadmap item.

### [HIGH-2] `/run/argonaut` created with world-traversable mode

- **File:** `src/notify.cyr` `notify_bind`.
- **Behaviour:** `sys_mkdir("/run/argonaut", 493)` — `493 = 0o755`,
  group + others get read+execute. Combined with the AF_UNIX
  socket inside (no peer-cred check, see **M1**), any local user
  who can `cd /run/argonaut` can reach the socket and inject fake
  sd_notify messages.
- **Severity:** **HIGH**. The notify socket is the trust anchor for
  service `READY=1` and `MAINPID=...` claims; spoofing it makes
  argonaut think a service is healthy / restart with a wrong PID,
  which cascades into watchdog and reaper logic.
- **Regression:** `tests/tcyr/notify_dir_perms.tcyr` — assert that
  `notify_bind` leaves `/run/argonaut` mode == 0o700.
- **Fix:** change `493` → `448` (0o700). The notify directory is
  argonaut-internal; no other process needs traversal.

### [MEDIUM-1] sd_notify: no `SO_PEERCRED` check

- **File:** `src/notify.cyr` `notify_try_recv` / `notify_parse`.
- **Behaviour:** every datagram on the AF_UNIX socket is parsed and
  applied. There is no `getsockopt(SO_PEERCRED)` check that the
  sender is the expected service PID (or in the expected pid set).
- **Severity:** **MEDIUM** — once **H2** is fixed, attackers without
  root can no longer reach the socket, dropping this to a
  defense-in-depth posture. Still worth implementing because the
  `NotifyAccess=` model is the canonical industry control here, and
  argonaut's threat model includes shared-tenancy hosts where the
  PID 1 surface is the last barrier.
- **Regression:** `tests/tcyr/notify_peer_cred.tcyr` — message from
  a sender whose PID isn't in the registered service set must be
  dropped, not parsed.
- **Fix:** new `notify_recv_authenticated(fd, expected_pids)` that
  uses `recvmsg` with `SCM_CREDENTIALS` + verifies the sender PID
  against `g_proc_table` PIDs. 1.4.0 ships the test + the helper
  function unwired (so consumers can adopt it); fully wiring it
  into `init_poll_health` is a 1.4.x patch follow-up.

### [MEDIUM-2] HTTP health-check path is a non-NUL-terminated `cstr` slice

- **File:** `src/health.cyr` HTTP_GET branch.
- **Behaviour:**
  ```
  if (slash_pos < url_len) {
      path = url_data + slash_pos;
      path_len = url_len - slash_pos;
  }
  ...
  str_builder_add_cstr(req_sb, path);
  ```
  `path` is a raw pointer into the URL Str's data buffer.
  `str_builder_add_cstr` reads until the first NUL byte —
  unspecified for a Str-data pointer, since Cyrius Str data
  buffers are not guaranteed NUL-terminated. The `path_len` is
  never used. The GET line therefore contains undefined trailing
  bytes whenever the URL has a path component.
- **Severity:** **MEDIUM** — the bug corrupts the GET line for any
  URL with a path. In practice this hits any non-`http://host/`
  health-check URL. May appear as 400-class responses, depending
  on what bytes follow `path` in the heap — a deterministic but
  configuration-dependent failure.
- **Regression:** `tests/tcyr/health_http_path.tcyr` — assert that
  the HTTP_GET request line for `http://127.0.0.1:8080/healthz`
  is exactly `GET /healthz HTTP/1.0\r\n...`, not corrupted.
  (Verifies via a stub socket recv.)
- **Fix:** copy the path bytes into a heap buffer and NUL-terminate,
  or use a length-bearing append:
  `str_builder_add_buf(req_sb, path, path_len)` — adding that
  helper if it doesn't exist.

### [MEDIUM-3] No generic-waitpid reaper for orphans (PID 1 path)

- **File:** `src/process_mgmt.cyr` `proc_table_reap`.
- **Behaviour:** only tracked PIDs are `waitpid`'d. As actual PID 1
  argonaut inherits orphans (services that re-fork, daemons whose
  parents exit, etc.); these accumulate as zombies.
- **Severity:** **MEDIUM**. When argonaut runs as a systemd-delegate
  (the current consumer state), the kernel routes orphans to systemd
  and this finding is dormant. When argonaut becomes true PID 1
  for AGNOS boot (the explicit roadmap goal), the gap manifests.
  Filing under MEDIUM because correctness for the stated mission
  requires the reaper.
- **Regression:** `tests/tcyr/reaper_orphans.tcyr` — fork a child
  that immediately fork-and-exits, mark argonaut as
  subreaper (`prctl(PR_SET_CHILD_SUBREAPER, 1)`); assert the
  generic reaper (new fn) collects the orphan within one tick.
- **Fix:** add `proc_table_reap_orphans()` that calls
  `sys_waitpid(-1, &status, WNOHANG)` in a loop until ESRCH/0, and
  wire it into `init_reap_services`. Subreaper enrol via prctl
  in `argonaut_init_new` for non-PID-1 contexts.

### [MEDIUM-4] PID file consumed without owner verification (CVE-2025-4598 class)

- **File:** `src/process_mgmt.cyr` `read_pid_file`.
- **Behaviour:** reads pid integer from file, calls
  `process_alive(pid)`, returns. No `fstat(fd)` to check that the
  pid file is owned by the expected uid / not world-writable; no
  cross-check that the PID's `/proc/<pid>/exe` resolves to the
  service binary.
- **Severity:** **MEDIUM** — defense-in-depth. The same class as
  CVE-2025-4598's PID-recycling primitive, applied to argonaut's
  pid-file consumption. An attacker who can write a pid file (or
  win a PID reuse race) can mislead argonaut about which process
  to monitor or signal.
- **Regression:** `tests/tcyr/pidfile_owner_check.tcyr` — pid file
  written by an unexpected uid (or world-writable mode) must be
  rejected.
- **Fix:** `read_pid_file` adds an `expected_uid` argument, fstats
  the open fd, rejects on mode-and-bits-other-write or
  uid mismatch. Long-term, argonaut should prefer `pidfd_open` +
  pidfd-equality checks over PID-based lookups, but that's a
  bigger lift and lands in a future minor.

### [LOW-1] `unlock_luks`: `mapped_name` not validated

- **File:** `src/edge_boot.cyr` `unlock_luks`.
- **Behaviour:** `mapped_name` (the device-mapper alias) is checked
  for non-empty only. Cryptsetup uses positional args so injection
  is bounded, but special characters in the name (slashes,
  quotes, control bytes) propagate unchecked into
  `/dev/mapper/<name>` paths.
- **Fix:** apply the same `[a-zA-Z0-9_-]{1,32}` predicate that
  yukti's `storage_eject` uses for sysfs basenames.

### [LOW-2] `verify_rootfs_integrity`: hex-charset of `root_hash` not enforced

- **File:** `src/edge_boot.cyr` `verify_rootfs_integrity`.
- **Behaviour:** `if (str_len(root_hash) != 64) { return 0; }` — but
  the bytes can be anything, including args-flag-like payloads.
  Cryptsetup positional argv catches this in practice; defense in
  depth says verify hex-only.
- **Fix:** add a `is_hex64(s)` predicate; call it on `root_hash`.

### [LOW-3] `fork_exec_service` does not `setsid` / close beyond fd 0

- **File:** `src/process_mgmt.cyr` `fork_exec_service`.
- **Behaviour:** child redirects stdin to `/dev/null` but inherits
  argonaut's stdout, stderr, and controlling TTY. Service log
  output is mixed into argonaut's; signals from the controlling
  TTY (e.g. `^C` if argonaut is foregrounded for testing) reach
  the children.
- **Fix:** `sys_setsid()` post-fork, plus an explicit `dup2`
  pipeline for stdout/stderr to the journal file or `/dev/null`.

### [LOW-4] `load_env_file` silently truncates files >8 KB

- **File:** `src/process_mgmt.cyr` `load_env_file`.
- **Behaviour:** `var buf = alloc(8192); var n = file_read_all(...,
  buf, 8192);` — files larger than 8192 bytes are read in full but
  parsing only sees the first 8 KB, with no warning.
- **Fix:** check `file_size > buf_size` via `lseek(fd, 0, SEEK_END)`
  and return a clear error if the env file exceeds the buffer.

### [LOW-5] HTTP response parser hardcodes HTTP/1.x byte positions

- **File:** `src/health.cyr` HTTP_GET branch.
- **Behaviour:** status code is read at offsets 9–11 of the response
  buffer, valid only for `HTTP/1.x ` (9-byte prefix + space). HTTP/2
  responses (rare for liveness probes but possible) are misparsed.
- **Fix:** find the first space, parse three digits after it.
  Defense in depth.

## Patra `json_build/6` shadow — upstream issue (not argonaut)

Discovered during the 1.3.0 toolchain bump (CHANGELOG 1.3.0 Fixed
section). `lib/patra.cyr:2097` defines a 6-arg `json_build(buf, max,
keys, vals, types, n)` that silently shadows the stdlib's 1-arg
`json_build(pairs)` — Cyrius has no overloading and emits no
arity-mismatch warning. **This is a patra-side issue, filed for
upstream fix.** Argonaut works around it by avoiding stdlib
`json_build/1`; CLAUDE.md and the Cyrius Conventions section flag
the gotcha. No action needed in 1.4.0.

## Verified-clean surface

These items were inspected and found correct on this audit pass:

- **`safe_cmd_*` API** — all process-execution paths in
  `src/{boot,services,edge_boot,tmpfiles,security,types}.cyr` use
  `safe_cmd_new` + explicit `safe_cmd_add_arg` argv; no shell
  string interpolation, no `sys_system` callers in `src/`.
- **`constant_time_eq_str`** is used for emergency-shell auth
  comparison (CHANGELOG 1.2.0 fix); password hashing uses sigil's
  SHA-256.
- **TCP health check** verifies `SO_ERROR` via `getsockopt` after
  poll-on-EINPROGRESS (1.2.0 fix); no false positives on
  `ECONNREFUSED`.
- **HTTP health check** rejects non-`http://` schemes (1.2.0 fix).
- **Audit chain** — argonaut's `audit_log_*` wrappers in
  `src/audit.cyr` are API-compatible with libro 2.0's breaking
  changes (`verify_chain` arity, nested-canonical-JSON hashing);
  details values are always strings, so the 2.0 hash semantics
  produce identical chain hashes for argonaut workloads.
- **No buffers ≥ 1 KB declared via `var buf[N]` inside a function**
  (the cyrius 5.x static-data gotcha). Smaller buffers
  (`var buf[256]` in `fleet_registration_from_system`,
  `var resp[512]` in `execute_health_check`, `var buf[64]` in
  `read_pid_file`) are present but their consumers either
  `str_clone` the result to heap before the next use or are
  single-shot per call frame — the static-data sharing isn't
  reachable in practice given argonaut's single-threaded model.
- **`var addr[110]`** in `notify.cyr` and `health.cyr` matches
  `sockaddr_un` (110) and `sockaddr_in` (16) sizes correctly.
- **No `str_from(...)` keys** passed to 5.x stdlib lookup helpers
  (`toml_get`, `map` lookups by cstr key). All `map_set` keys are
  bare cstr literals.

## Re-audit triggers

Run another full P(-1) audit when **any** of:

- argonaut graduates from systemd-delegate to true PID 1 (the
  reaper / signal-handling surface changes radically)
- a libro / patra major bump lands (canonical hashing or storage
  semantics changing)
- cyrius adopts function overloading (the patra `json_build` class
  collapses or expands)
- a new init-system CVE class lands in CVE.org's NVD against
  systemd / OpenRC / s6 / runit

Next routine audit: 2026-Q3 (or release time, whichever first).
