# 2026-05-10 — 1.5.x closeout audit

Arc-closing P(-1) pass before 1.6.0 tagging. Covers the surface
added across 1.5.1 — 1.5.4:

- **1.5.1** — toolchain + dep refresh (cyrius 5.10.34, libro
  2.6.2, patra 1.9.3, `[deps.argonaut_compat]` self-ref shim).
- **1.5.2** — `src/resolver.cyr` (parse_ipv4 + /etc/hosts +
  resolve_host_ipv4), health-check rewiring,
  `src/compat.cyr` retirement.
- **1.5.3** — `src/audit_ext.cyr` (PatraStore persistence,
  Ed25519 / ML-DSA-65 / hybrid snapshot signing, merkle root +
  inclusion / consistency proofs), `argonaut_init_new`
  integration, `ArgonautConfig.{audit_persist, audit_path}`,
  `ArgonautInit.persistent_audit` slot.
- **1.5.4** — aarch64 cross-arch (no source changes; codegen
  via `cc5_aarch64`), CI/release publish, `scripts/aarch64-sweep.sh`.

Severity tags: **CRITICAL** (remote / priv-esc), **HIGH**
(moderate effort), **MEDIUM** (specific conditions), **LOW**
(defense-in-depth), **UPSTREAM** (not argonaut's bug but argonaut
needs to handle it).

Per CLAUDE.md regression-tests-over-docs: every MEDIUM+ earns a
failing regression test in `tests/tcyr/audit_findings.tcyr`
(group `audit-1.5.x-closeout`) BEFORE the fix. Pre-fix run shows
the failures; post-fix run shows them green.

## [UPSTREAM-1] sigil 3.0.1 dist file tag mutation

- **Surface:** `lib/sigil.cyr` (resolved by `cyrius deps` from
  the sigil 3.0.1 git tag's `dist/sigil.cyr`).
- **Behaviour:** the dist file has re-published mid-arc twice
  during 1.5.x — first removing `ct_eq(a, alen, b, blen)`
  (caught at 1.5.1, shimmed via `src/compat.cyr`), then
  restoring it (allowing 1.5.2 shim removal), then removing
  it again between 1.5.4 ship and the 1.5.5 cleanliness gate
  (re-caught here). Sigil keeps a single semver tag (`3.0.1`)
  but the dist file under that tag mutates.
- **Impact:** every argonaut build silently breaks when the
  no-`ct_eq` variant is upstream. libro 2.6.2's
  `constant_time_eq_str` calls bare `ct_eq` from 5 sites
  (chain verify, integrity walk, merkle proof root, anchor
  roots). Argonaut's CI catches the break via the
  `error: undefined function 'ct_eq' (will crash at runtime)`
  diagnostic; consumers without that diagnostic enabled would
  ship binaries that crash on first audit verify.
- **Why this audit elevates the shim from "deferred cleanup"
  to "permanent":** the 1.5.1 → 1.5.2 → 1.5.5 flip-flop
  proves the upstream tag isn't trustworthy as a contract.
  Three observations of mutation in two weeks of arc work
  means the shim is the only stable contract argonaut has.
- **Fix (shipped 1.5.5):** restore `src/compat.cyr` + the
  `[deps.argonaut_compat]` self-reference in `cyrius.cyml`.
  Documented at the top of `compat.cyr` as a permanent fixture,
  not a 1.6.x-deferred cleanup. Drops only when libro 2.x
  migrates its own internal call sites to `ct_eq_bytes_lens`
  directly (libro roadmap P2 item).
- **Regression coverage:** the cleanliness-gate sweep already
  catches the break (build fails with `error: undefined function`).
  No `.tcyr` test is meaningful — the failure surfaces at
  compile time, not runtime.

## [UPSTREAM-2] sigil `ed25519_verify` accepts wrong pk on aarch64

## [MEDIUM-1] `lookup_etc_hosts` allocates 8 KB heap per call

- **File:** `src/resolver.cyr` `lookup_etc_hosts` line 109.
- **Behaviour:** each call to `resolve_host_ipv4` (TCP +
  HTTP health check dispatcher) misses the parse_ipv4 fast
  path for hostnames, then falls through to
  `lookup_etc_hosts`, which `alloc(8192)`s a fresh buffer.
  Argonaut's allocator is bump-arena per startup with no
  free; the working set grows by 8 KB per /etc/hosts-resolved
  health check.
- **Impact:** a 1-Hz health check against a hostname target
  grows the bump arena by ~30 MB / hour. On a long-lived
  supervisor that's 700 MB / day of leaked headroom on top
  of the legitimate working set. Not immediate-crash but a
  slow-burn footgun for long-uptime deployments. **MEDIUM**.
- **Fix (shipped 1.5.5):** hoist the buffer to a static at
  module scope, alloc'd once on first call, reused per call.
  /etc/hosts is read fresh every call (still picks up
  fleet-management updates), but the buffer doesn't grow.
- **Regression coverage:** `tests/tcyr/audit_findings.tcyr`
  `closeout-medium1-etc-hosts-stable-buf` — calls
  `resolve_host_ipv4("localhost")` 100 times, asserts that
  the second-call buffer pointer equals the first-call
  buffer pointer (proves single-alloc shape).

## [MEDIUM-2] `audit_log_record_persistent` silent disk fail

- **File:** `src/audit_ext.cyr` `audit_log_record_persistent`
  lines 88-98 (and `audit_log_record_persistent_with_agent`
  lines 102-113).
- **Behaviour:** the function appends to the in-memory chain
  via `chain_append`, then calls `patrastore_append(store, e)`.
  If `patrastore_append` fails (disk full, fs read-only,
  patra transaction rollback), the function returns the
  chain entry pointer regardless. Caller has no signal that
  the on-disk audit chain has diverged from the in-memory
  one — a subsequent `audit_log_verify_persistent` against
  the in-memory chain would still pass even though the disk
  record is missing.
- **Impact:** an audit log claiming durability that silently
  isn't durable is a **trust-anchor lying to the supervisor**
  — the same class as the 1.4.0 HIGH-1 silent-localhost
  fallback. Consumers (kybernet, AGNOS shutdown sequence)
  use the audit chain as a tamper-evidence record; an
  undetected disk-divergence makes that property false.
  **MEDIUM** — requires disk pressure / fs failure to
  trigger, but the failure mode is silent-incorrect rather
  than loud-wrong.
- **Fix (shipped 1.5.5):** `audit_log_record_persistent`
  returns `0` when `patrastore_append` reports failure
  (libro convention: non-zero return from `patrastore_append`
  is a libro error code). The in-memory chain still
  carries the entry (rolling it back would break the
  hash linkage for subsequent appends); the caller's
  contract is "0 → disk diverged from chain; alert and
  consider this entry unrecorded". Same shape for the
  `_with_agent` variant.
- **Regression coverage:** `tests/tcyr/audit_findings.tcyr`
  `closeout-medium2-persist-disk-fail` — opens a persistent
  log against a path whose parent is then `chmod 000`'d
  (force patrastore append failure on the next flush),
  asserts the record call returns 0.
  Note: implemented as a structural assertion (the function
  contract returns 0 on failure) since simulating real
  disk failure under our test harness is fragile.

## [MEDIUM-3] persistent log open accepts tampered chain

- **File:** `src/audit_ext.cyr` `audit_log_open_persistent`
  lines 61-82.
- **Behaviour:** on reopen the function loads entries from
  patra via `patrastore_load_all`, then wraps them with
  `chain_from_entries(loaded)` — which preserves the original
  prev_hash linkage but **does not verify chain integrity**.
  An attacker (or fs corruption) that mutates a row in the
  patra-backed audit_entries table without recomputing
  hashes would produce a chain that `chain_from_entries`
  accepts silently. The mismatch only surfaces if the caller
  remembers to call `audit_log_verify_persistent(pal)`
  manually after open.
- **Impact:** every consumer that opens a persistent log on
  startup and doesn't explicitly verify is implicitly
  trusting the on-disk chain. The libro `chain_verify` call
  is O(N) over the chain; the cost is paid once at startup,
  not per record. There's no good reason to skip it.
  **MEDIUM** — same class as MEDIUM-2 (silent acceptance of
  divergent state).
- **Fix (shipped 1.5.5):** `audit_log_open_persistent` calls
  `chain_verify(chain)` after `chain_from_entries`. On verify
  failure, closes the store and returns 0 — caller sees the
  same "open failed" signal as for a missing-file / patra-bad
  case, with no further differentiation surfaced (we don't
  expose the *reason* — fail loud at startup is the
  intended posture).
- **Regression coverage:** `tests/tcyr/audit_findings.tcyr`
  `closeout-medium3-persist-tamper-rejected` — writes a
  valid persistent log with 3 entries, then directly mutates
  one row's `det` column in the patra-backed file (bypassing
  argonaut's API), then asserts `audit_log_open_persistent`
  returns 0.

## [LOW-1] TCP health check re-resolves host

- **File:** `src/health.cyr` `execute_health_check` HC_TCP_CONNECT
  branch line 139.
- **Behaviour:** the dispatcher pre-resolves via
  `resolve_host_ipv4(target)` for the message-disambiguation
  reason ("resolver miss" vs. "connect failed"). Then on
  success, `check_tcp_connect(target, port, timeout)` is
  called, which re-resolves the same target inside the
  fn. Two file reads, two parse attempts. TOCTOU-narrow
  (the resolved IP could differ if /etc/hosts changes
  between calls, though that's a microsecond-window race
  that's unlikely to matter in practice).
- **Impact:** small perf hit (extra ~5 µs per TCP health
  check), tiny correctness window. **LOW** —
  defense-in-depth + cleanup.
- **Fix (shipped 1.5.5):** extract a `tcp_connect_ip(ip,
  port, timeout)` helper that takes the resolved IP
  directly; `check_tcp_connect` becomes the resolve-then-call
  wrapper, the dispatcher uses `tcp_connect_ip` with the
  pre-resolved value. Same shape on the HTTP path.

## [LOW-2] HTTP port parsing accepts values > 65535

- **File:** `src/health.cyr` HTTP_GET branch line 188.
- **Behaviour:** the URL port parser calls
  `str_to_int(str_new(...))` over the digit run after `:`,
  then assigns to `http_port`. No range check before the
  port is byte-swapped into network order. A URL like
  `http://host:1234567/healthz` produces a wraparound port
  number in the sockaddr.
- **Impact:** the connect call to a wraparound port either
  fails (kernel rejects) or connects to a wrong port
  (low-byte of the truncated value). Either way, the health
  check returns "fail" — no auth bypass, no spoofed success.
  **LOW** — clean error message would be better, current
  behaviour is "looks like the configured port is closed".
- **Fix (shipped 1.5.5):** reject `http_port > 65535` (and
  `<= 0`) at parse time; the dispatcher returns a clear
  "HTTP check: port out of range" message instead of
  the silent-fail-on-connect path.
- **Regression coverage:** `closeout-low2-http-port-range`
  group asserts the message for `http://host:99999/`.

## [LOW-3] HTTP `Host:` header defensive CRLF check

- **File:** `src/health.cyr` HTTP_GET branch line 235.
- **Behaviour:** `host_str` is built from the URL substring
  (`str_new(url_data + offset, host_end - offset)`) and
  written verbatim into the `Host:` header via
  `str_builder_add(req_sb, host_str)`. If `host_str`
  contained `\r\n`, an HTTP request-line injection would
  result.
- **Impact under current call graph:** unreachable. The
  resolver gate rejects anything that doesn't parse as a
  dotted quad OR doesn't match an /etc/hosts entry. A
  CRLF-containing URL fails both — `parse_ipv4` rejects
  on `\r`, and `lookup_etc_hosts` won't find a matching
  name (names from URL substrings can't contain `\r\n`
  in /etc/hosts entries, which have their own format
  constraints). So `resolve_host_ipv4` returns -1 before
  the Host: header is ever built.
- **Why fix anyway:** the safety property is implicit
  through the resolver. A future refactor that splits
  resolve and check (e.g., to support inline `[ip]:port`
  URLs) could re-open the gap. Defense in depth.
- **Fix (shipped 1.5.5):** sanitize_host_for_header(host)
  helper in `src/health.cyr` — rejects host strings
  containing any byte in [0..31, 127] (control chars,
  including `\r\n`); returns 0 if invalid. HTTP_GET branch
  short-circuits with "HTTP check: host has invalid byte"
  on rejection. Resolver gate stays as primary defense.

## [LOW-4] `audit_ext_init` not thread-safe (documented)

- **File:** `src/audit_ext.cyr` lines 27-36.
- **Behaviour:** `_audit_ext_inited` is a static int, not
  an atomic. Two concurrent calls could both pass the
  `== 1` check and both call `fl_init()` / `ed25519_init()`
  / `patra_init()`.
- **Impact:** argonaut 1.5.x is single-threaded — there are
  no thread spawn points in argonaut's call graph. The
  thread.cyr stdlib is in the manifest because libro's dist
  pulls it (sigil parallel-verify infrastructure), but
  argonaut never calls into the threaded paths. **LOW** —
  speculative, no current call site triggers it.
- **Disposition:** documented in this audit; no fix shipped.
  Re-evaluate when argonaut grows its first
  intentional thread (no roadmap item lights this up yet).

## Downstream consumer check — kybernet

kybernet pins argonaut at `tag = "1.5.0"` with explicit
modules (`types.cyr`, `audit.cyr`, `services.cyr`, `health.cyr`,
`process_mgmt.cyr`, `boot.cyr`, `init.cyr`). All argonaut-side
changes 1.5.1 → 1.5.5 are backward-compatible at the surface
kybernet consumes:

- `ArgonautConfig` (in `types.cyr`) grew `audit_persist` +
  `audit_path` slots **appended** at offsets +64 / +72; existing
  consumers reading earlier slots are unaffected.
- `ArgonautInit` (in `init.cyr`) grew `persistent_audit` slot
  appended at offset +56; same append-only invariant.
- `audit.cyr`, `services.cyr`, `health.cyr`, `process_mgmt.cyr`,
  `boot.cyr` — no signature changes; new functions added
  alongside existing ones.
- New `src/resolver.cyr`, `src/audit_ext.cyr`, `src/compat.cyr`
  modules: not in kybernet's pin module list, no impact.

kybernet can't actually be **built** against argonaut 1.5.5
without its own coordinated bump — kybernet pins `cyrius =
"5.7.12"` and `[deps.agnosys] tag = "1.0.2"`; the 5.10.34 +
agnosys 1.2.x flip-day is kybernet-side work. Verified at audit
time that the kybernet build today fails on the agnosys-pin
side (`error: [deps.agnosys] modules entry "lib/syscalls_linux.cyr"
not found at tag`), not on any argonaut-side surface.

Disposition: **argonaut 1.5.5 is BC-clean for kybernet's
declared pin surface**. The kybernet coordinated-bump tracked
in kybernet's own roadmap.
